1. Auditing The XML OSM File, And Problems Encountered

                                                

The tasks to explore the OSM XML file were:

  • Exploring the tags names and their total number of occurrences, and extracting the limits defined in the file.
  • Auditing regular expressions.
  • Total number of unique users that have contributed to the map.
  • Auditing street names.

Auditing and Correcting Street Names

                                                

Initially, improving_street_names output of interest was auditing street types. Given this information, the first step was to define the expected types which prove to be a relatively large set, including types such as 'Alley', 'Crescent', 'Mews', 'Walk', and others more common. There were also cases in which the type was missing because the street is only know by its name, without a type, such as the case with 'Kingsway'; in these cases it was decided to left the street without a type. There were aslo a couple of cases in which 'Vancouver' was added at the end of the street name unnecessarily, in these cases the name of the city was removed to avoid redundancy and to keep consistency. Types such as 'Mall' and 'Terminal' seem to point to a different error, that is to say, not an error in style such as abbreviation, but an error in the type of the tag (instead of 'way', it may be a 'node' element); however, these elements are left as such since further manipulation is needed.

                                                

With the expected types defined, it was decided that the mapping used woukld fix abbreviations, and eliminate the name of the city of Vancouver in the street name field where necessary. The OSM file was processed, and a new xml file; my_vancouver.xml - file size: 85,6 GB -, was generated with the correct street names.

                                                

Subsequently the XML shape of the data was transformed into a nested dictionary JSON-like shape in data_xml_into_dicts and loaded in my_vancouver.xml.json - file size: 106,9 GB -. Finally, the JSON file, with the corrections was loaded to MongoDB using the mongodb_collection code, creating the 'OSM' db, and the 'vancouver' collection.

Comparing this value to the one obtained auditing the tags of the original OSM file, and having in mind that the JSON file used to create the MongoDB collection only contains 'ways' and 'nodes', the addition; 67187 (number of ways) + 387599 (number of nodes) = 454786, is consistent with this result.

> db.vancouver.find({"type":"way"}).count()

Top 5 'cuisine' in a given area

                                                

To run this query, first the bounds for the area were determined by using this coordinates site to be between longitude -123.145 and -123.1. This area contains the downtown.

> db.vancouver.aggregate( [{ $match : {"amenity" : "restaurant"}},

3. Additional Ideas

Possible 'type' misnomer

Initially, the fact that there are street types such as "Mall" and "Terminal", were thought to be hints that a 'node' was being classified as a 'way'. However, this turn out not to be the case. The problem was that some nodes are located in Malls and in Station Terminal, and the address:street field is used to specify the name of the Mall.

An alternative way to specify this address would be to leave the address of the Mall as the address of the node, adding a field to specify the position of the node within the Mall.

Missing Information

In the restaurants queries it was found that several amenities classified as restaurants don't have a street field in their addresses, and also don't have a cuisine field.

                                                

The fact that in both cases, the majority of elements have "null" values, indicates that there is a important issue with missing information.

Location Queries

The downtown restaurants query could also have been done using the MongoDB 2D Geospatial Indexing and Queries, setting the $near operator to a spot in the middle of downtown.

Conclusion

The exploration done in this exercise of the OSM Vancouver map was only the very tip of the iceberg. Several tasks to analyze and improve this data are at hand. There are several other amenities in which field values may be missing; however, auditing completeness, as auditing accuracy does too, needs some kind of trusted reference data which is not easy to gather.

                                                

Also, it is still not clear the actual reason for the discrepancy in number of nodes and ways in the OSM file and the MongoDB Collection. For this, auditing field values validity may be necessary using for instance some regular expressions insights.
